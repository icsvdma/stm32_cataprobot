# 不具合報告書: STM32 SPI スレーブ受信によるメインループ停止

| 項目 | 内容 |
|---|---|
| **報告日** | 2026-04-04 |
| **対象MCU** | STM32G473RCTx |
| **対象通信** | SPI1 Slave（ESP32 Master → STM32 Slave） |
| **ステータス** | **暫定対処済み（ポーリング方式で動作確認）** |
| **影響範囲** | メインループ完全停止 → IWDG タイムアウト → ウォッチドッグリセット |

---

## 1. 現象概要

ESP32 から SPI1 経由でコマンドパケットを送信すると、STM32 のメインループが停止し、約 8 秒後に IWDG（独立ウォッチドッグ）リセットが発生して DFU モードに遷移する。

### 1.1 発生条件

- ESP32 が SPI Master として 2 秒間隔でパケット送信
- STM32 が SPI1 Slave（DMA + EXTI4 割込み方式）で受信

### 1.2 観測された症状

| 症状 | 詳細 |
|---|---|
| LED0 (PC10) 点滅停止 | メインループが回っていない |
| LED1 (PC11) 点灯 | 初期化は正常通過 |
| LED4 (PC14) 点灯 | 少なくとも1フレーム受信後に停止 |
| 約 20 秒後に DFU モード | IWDG リセット → BOOT0 設定により DFU 起動 |

---

## 2. 調査経過

### 2.1 切り分けテスト結果

段階的にモジュールを有効化し、どの初期化関数が停止を引き起こすか特定した。

| テスト | 構成 | 結果 |
|---|---|---|
| **① ベアメタル GPIO** | `MX_GPIO_Init()` のみ、NOP ループ LED 点滅 | **LED0 点滅 ✓** |
| **② 全 HAL Init** | MX_GPIO/DMA/SPI1/IWDG 全て有効 | **LED0 点滅 ✓**（全 Init 通過） |
| **③ UART 追加** | ② + `MX_USART2_UART_Init()` | **停止 ✗**（不具合 A） |
| **④ SPI のみ** | ② + `DebugLed_Init()` + `SPI_Slave_Init()` | **LED0 約 1 秒で停止 ✗**（不具合 B） |
| **⑤ SPI 除外** | ② + `DebugLed_Init()` のみ | **LED0 点滅 ✓** |

### 2.2 特定された不具合

---

### 不具合 A: UART 未接続時のハング

| 項目 | 内容 |
|---|---|
| **原因** | `MX_USART2_UART_Init()` が IDLE LINE 割込み + DMA 受信を有効化。UART RX ピン (PA3) が未接続（フローティング）のため、ノイズで連続的に IDLE 割込みが発火し CPU を専有 |
| **影響** | メインループに制御が戻らない |
| **対処** | `MX_USART2_UART_Init()` 呼出しをコメントアウト |
| **恒久対策** | UART 使用時は RX ピンにプルアップ抵抗を実装するか、UART 接続前は Init しない制御を追加 |

---

### 不具合 B: SPI DMA/EXTI 割込み方式の競合によるハング

| 項目 | 内容 |
|---|---|
| **根本原因** | SPI1 DMA 受信 + EXTI4（NSS 立ち上がり割込み）の組み合わせで、ISR 間の競合が発生しメインループがブロックされる |
| **発生メカニズム（推定）** | 下記参照 |
| **影響** | メインループ停止 → IWDG リセット → DFU モード遷移 |

#### 推定メカニズム

```
1. ESP32 が CS=LOW にして SPI 送信開始
2. DMA1_Channel2 が MOSI → s_rx_buf への転送を開始
3. ESP32 が CS=HIGH にして送信完了
4. EXTI4 割込み (NSS 立ち上がり) が発火
5. EXTI4 ハンドラ内で SPI_Slave_NssRisingCallback() が呼ばれ
   → パケット解析 + 新規 DMA 受信開始
6. DMA 転送中に新しいフレームが到着、または
   DMA 完了割込みと EXTI4 割込みが同時に発生
7. HAL ライブラリ内部の状態マシンが不整合 → 
   SPI ペリフェラルが BUSY のまま解除されない
8. 以後、SPI 操作が全てブロック
```

#### 試行した修正と結果

| 試行 | 内容 | 結果 |
|---|---|---|
| **修正 1** | `SPI_Slave_Poll()` 内で EXTI4 を一時無効化 | **停止 ✗** |
| **修正 2** | 修正 1 + DMA1_Ch2/SPI1 割込みも一時無効化、`RxCompleteCallback` をフラグ方式に変更 | **停止 ✗** |
| **修正 3** | DMA/EXTI を完全廃止、レジスタ直接ポーリング方式に全面書き換え | **動作 ✓（ただし追加修正が必要だった）** |

---

### 不具合 C: ポーリング方式での RXNE 無限ループ

| 項目 | 内容 |
|---|---|
| **原因** | `while (spi->SR & SPI_SR_RXNE)` のループに上限がなく、SPI がデータを連続受信する環境ではループから抜けられない |
| **影響** | メインループが `SPI_Slave_Poll()` 内でブロック → IWDG リセット |
| **対処** | 1 回の呼出しで最大 8 バイトに制限 + タイムアウトによるフレーム完了判定を導入 |

---

### 不具合 D: DebugLed_Update() の SPI ノイズ依存

| 項目 | 内容 |
|---|---|
| **原因** | ESP32 停止中も SPI 信号線のフローティングノイズにより RXNE が常に立ち、`s_rx_idx > 0 && cs_high` 条件で `GPIOC->BSRR = PIN_DBG_LED4` が実行され続ける。これにより `memset` が毎ループ実行されるなどオーバーヘッド増加、かつ `DebugLed_Set()` 経由の BSRR 書込みと診断用 BSRR 直接書込みが干渉する可能性 |
| **影響** | LED0 点滅が見えない（DebugLed_Update 未呼出 or BSRR 干渉） |
| **対処** | LED0 を `DebugLed_Update()` 依存からレジスタ直接 ODR toggle に変更、500000 回 NOP ディレイ挿入 |
| **恒久対策** | SPI 信号線にプルアップ/プルダウン抵抗を実装してフローティング防止 |

## 3. 最終的な対処内容

### 3.1 spi_comm.c — 全面書き換え

| 変更前 | 変更後 |
|---|---|
| DMA (HAL_SPI_Receive_DMA) | レジスタ直接 RXNE ポーリング |
| EXTI4 NSS 立上り割込み | GPIOA IDR による CS 状態読取り + タイムアウト |
| ISR 内パケット解析 | メインループ内パケット解析 |
| 無制限 RXNE 読出し | 1 回最大 8 バイト制限 |

### 3.2 main.c — 変更点

| 項目 | 変更 |
|---|---|
| `MX_USART2_UART_Init()` | コメントアウト（UART 未接続のため） |
| LED0 点滅 | `DebugLed_Set()` → レジスタ直接 `GPIOC->ODR ^= PIN_DBG_LED0` |
| メインループ | `SPI_Slave_Poll()` → `HAL_IWDG_Refresh()` のみ |

### 3.3 stm32g4xx_it.c — 変更なし

EXTI4_IRQHandler は残っているが、`SPI_Slave_Init()` 内で `HAL_NVIC_DisableIRQ(EXTI4_IRQn)` により発火しない。

## 7. 教訓

1. **STM32 HAL の SPI Slave + DMA + EXTI は競合リスクが高い** — HAL の内部状態マシンが Master 指向で、Slave の非同期 NSS 変化に対して脆弱
2. **未接続ペリフェラルの Init は危険** — UART RX がフローティングだとノイズによる連続割込みで CPU 専有
3. **ポーリング方式では読出し上限が必須** — `while (RXNE)` は SPI データが連続する環境では無限ループになる
4. **段階的 LED マーカーによるデバッグは有効** — JTAG/SWD なしの環境で問題箇所を特定できた
